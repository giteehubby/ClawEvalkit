Scoring System Comparison

PinchBench — Auto + LLM
  • 每个任务有明确的 grading criteria checklist
  • 内置 Python 评分函数（自动化检查）
  • 部分任务使用 LLM judge 进行语义评估
  • LLM judge 可能引入主观偏差
AgentBench — 4-Layer System
  • L0 (20%): 文件存在/格式/内容自动检查
  • L1 (35%): 工具调用次数/规划比/错误数量
  • L2 (20%): 行为分析（自评）
  • L3 (25%): 输出质量（自评）
Claw-Bench — Pytest Weighted
  • 每个任务 12-28 个 pytest checkpoint
  • Weight 3: 核心正确性（文件存在、值正确）
  • Weight 2: 标准质量（格式校验）
  • Weight 1: 加分严格性（无占位符、一致命名）
TRIBE-INC — Pass/Fail
  • PASS: 输出符合预期标记
  • WARN + PASS: 通过但有 caveat
  • FAIL: 未满足最低要求
  • CRITICAL FAIL: 核心功能损坏
SkillsBench — Three-condition Pytest
  • 确定性 pytest 验证器（非 LLM judge）
  • Oracle solution 必须 100% 通过
  • 三条件对比：vanilla / curated / self-gen
  • Normalized gain = (skills - vanilla) / (1 - vanilla)
  • Docker 容器隔离，每任务独立环境

Key Finding: Bedrock vs OpenRouter

Kimi K2.5 Bedrock Timeout Bug (Proven)

10 次相同请求测试：6 次成功，4 次超时（60% 成功率）。多次 tool use 成功率呈指数衰减：60%^n。

原因：AWS Bedrock Converse API 在处理 tool results 时出现间歇性 stream 挂起，不是模型 bug。

解决方案：使用 OpenRouter 替代 Bedrock。
