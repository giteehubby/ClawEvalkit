# 诊断与修复 OpenClaw 类智能体：统一评估、失败分类与轻量级配方

## 摘要

OpenClaw 类开源智能体已在多步骤工具使用和推理任务中获得了广泛部署，然而它们的失败模式仍然未被充分理解。之前的工作要么使用昂贵的训练配方优化这些智能体，要么报告聚合的基准测试分数而不诊断*什么*出了问题以及*为什么*。我们提出了一项统一研究，从评估到诊断再到修复形成闭环。**我们的核心主张是：Open-source agent failures are often structural, not parametric.** 也就是说，相当比例的智能体失败并非来自 base model 缺乏任务知识，而来自能力到调用之间的激活鸿沟（**H1**）、状态表示的抽象失败（**H2**）、或控制闭环的断裂（**H3**）。我们构建了固定评估框架，并在两个智能体基准测试上建立了包含五类主要失败的**系统化失败分类体系**（A-E）。通过对 4-5 个持续失败任务的案例分析，我们识别出结构性失败模式：E 类失败（无效的技能激活或组合）占任务的 13-17%，需要根据技能是缺失还是只是未被充分利用来选择不同的修复策略；AgentBench-OC 上的记忆域在所有模型检查点上持续得分 50.0，表明存在结构性 C 类失败。**我们进一步证明，training-free 干预和 SFT 修复的是正交 failure modes（H4）：前者主要修复结构性失败（A/B/D/C），后者主要修复知识/技能激活失败（E）。** 我们的发现为从业者提供了一个原则性决策树：从根本上区分"模型不会"还是"系统层激活/控制失败"，选择对应成本的修复策略。

## 1 引言

OpenClaw 类智能体——使用大型语言模型进行规划、调用工具、浏览网页和综合答案的开源通用智能体——已成为多步骤推理和研究任务的标准架构。尽管它们被广泛采用，但一个根本问题仍未得到充分回答：当这些智能体失败时，*什么*系统性地出了问题，以及*哪些*失败可以在什么成本下被修复？

改进智能体系统的工作之前遵循两条几乎脱节的路径。第一条路径专注于*基于训练*的方法：应用偏好优化、强化学习从人类反馈中学习，或在精选轨迹上进行监督微调。这些方法有效但昂贵：它们需要大量偏好数据、显著的训练算力，并且可能在智能体已经擅长的任务上产生负干扰。第二条路径专注于*推理时*配方：添加记忆缓冲区、实现规划器-执行器分离，或设计多智能体协作协议。这些方法轻量，但通常在隔离环境中评估，没有与基于训练的替代方案进行系统比较，也没有严格诊断每个配方实际修复了哪些失败类型。

**本文的核心主张是：Agent failures are often misdiagnosed as model weakness, while many are actually system-level activation or control failures.** 我们通过第一性原理方法研究这个问题：*我们能将智能体失败诊断为少量稳定、可解释的类别吗？轻量级无训练配方能否以接近零成本修复大部分 A-D 类失败？而小数据监督微调在哪里提供最大的边际价值？*

## 2 统一评估框架

我们的第一个贡献是一个可重现的评估框架，用于在同等条件下比较智能体条件。该框架围绕三个原则设计：**固定版本**（每个基准测试锁定到特定的 git 提交或版本）、**统一日志**（每次运行生成包含模型元数据、配置和成本核算的清单）以及**基准测试内配对比较**（所有条件在相同的任务实例上评估，实现配对统计分析）。

### 2.1 基准测试套件

我们评估涵盖多样化智能体任务配置文件的两个基准测试：

- **AgentBench-OpenClaw (AB-OC)：** 通用多步骤工具使用基准测试，包含 40 个任务横跨 7 个域（文件创建、研究、数据分析、多步骤、记忆、错误处理、工具效率）。任务平均 6-10 步，包含多次工具调用。我们使用固定版本的 OpenClaw skill 发布版和标准任务分割。

- **PinchBench (PB)：** 混乱的真实世界生产力任务基准测试，包括日历管理、邮件分类、市场研究、代码生成和跨会话记忆。任务涉及中等工具使用（2-5 次调用）但需要大量状态管理。我们使用固定版本的 v0.9 发布版，包含全部 23 个任务。

### 2.2 智能体条件

我们比较以下条件，所有条件均在 nanobot OpenClaw 智能体运行时上使用 doubao-seed-1.8 作为基础模型进行评估：

1. **Base：** 未修改的 OpenClaw 智能体，贪婪解码，温度 0.0，最多 40 步。

2. **+Memory：** 基础智能体配备任务本地情景记忆缓冲区，记录中间事实、文件路径和经验证的结论。记忆在每次工具调用后写入，在每次新工具调用前读取。

3. **+Control：** 基础智能体配备计划优先提示（acting 前明确生成步骤计划）、有界重规划（每 5 步重规划）和失败触发反思（如果工具返回错误，在重试前生成明确诊断）。

4. **+Collab：** 基础智能体配备最小双角色结构：一个*执行器*调用工具，一个*验证器*根据任务目标检查中间结论。角色通过共享消息缓冲区通信，验证器可以请求回滚到之前的检查点。

5. **+ProcSupport：** 基础智能体配备按需访问每个基准测试域的结构化程序支持卡片。卡片包含分步领域程序，通过任务开始时的语义相似性检索。

6. **+BestTrainFree：** 根据初步分析确定的最佳无训练配方组合（记忆 + 控制）。

7. **SFT-100、SFT-300、SFT-1000：** 分别在 100、300 和 1000 个修正轨迹上训练的 LoRA-SFT 变体，从基础条件的失败中采样。

### 2.3 成本核算和指标

每次运行记录：总提示 token、完成 token、工具调用、步数、wall-clock 时间和估计 API 成本。

我们的主要指标是**基准测试内配对增量**：

$$\Delta_b(c) = \frac{1}{|\mathcal{T}_b|}\sum_{t\in\mathcal{T}_b}[\mathrm{score}_c(t) - \mathrm{score}_{\mathrm{Base}}(t)]$$

我们还跟踪**负干扰率**（NIR）和每个失败类别的**类别修复率**（CRR）：

$$\mathrm{CRR}_k(c) = P(\mathrm{score}_c(t) = 1 \mid \mathrm{score}_{\mathrm{Base}}(t) = 0, \mathrm{cat}(t) = k)$$

## 3 失败分类体系

在尝试修复智能体失败之前，我们必须首先诊断*什么*失败了以及*为什么*。我们通过分析基础条件失败案例定义了一个五类失败分类体系（A-E）。

### 3.1 分类体系定义

**[类别 A] 任务理解 / 规划漂移**

*定义：* 智能体未能正确解释任务要求、约束或目标，或在多步骤执行过程中逐渐偏离核心目标。

*候选修复：* 计划优先提示、明确子目标跟踪、周期性重规划。

**[类别 B] 工具 / 环境落地失败**

*定义：* 智能体正确理解*什么*它想要做，但未能正确地将意图映射到执行环境所需的正确工具、工具参数、文件路径或输出格式。

*候选修复：* 环境就绪检查、工具模式落地、错误诊断重试。

**[类别 C] 记忆 / 状态管理失败**

*定义：* 智能体未能稳定地维护任务上下文、中间事实、完成步骤生成的工件或跨交互历史的先前搜索结果。

*候选修复：* 任务本地情景记忆缓冲区、结构化状态跟踪器、工件注册表。

**[类别 D] 验证 / 恢复缺陷**

*定义：* 智能体缺乏自我检查、错误检测或恢复能力。它在错误后继续而不纠正，进入无效循环，或未能诊断工具调用返回意外结果的原因。

*候选修复：* 内置验证器/评论器、失败触发反思、明确恢复策略。

**[类别 E] 技能利用缺陷**

*定义：* 智能体拥有必要的技能（无论是内置的还是通过技能基础设施提供的），但未能有效激活、组合或利用它们来完成任务。

*候选修复：* 结构化程序支持卡片、基于技能的示例、在修正的程序演示上进行小数据 SFT。

**表 1：失败分类体系定义和候选修复。**

| 类别 | 描述 | 诊断证据 | 候选修复 |
|------|------|---------|---------|
| **A. 任务理解** | 智能体误解任务目标或执行中漂移 | 遗漏约束、追求错误子目标、答案跑题 | 计划优先、重规划、子目标跟踪 |
| **B. 工具落地** | 错误的工具/参数/执行上下文格式 | 错误工具选择、格式错误参数、不兼容输出格式 | 前置检查、模式落地、诊断重试 |
| **C. 记忆/状态** | 无法跨步骤维护上下文 | 重复搜索、丢失事实、文件/路径混淆 | 情景记忆、工件注册表、状态跟踪器 |
| **D. 验证** | 错误后无自我检查或恢复 | 忽略错误、循环、无恢复计划 | 验证器/评论器、失败反思、有界重试+诊断 |
| **E. 技能利用** | 有技能但未能有效激活/组合 | 错误的技能选择、技能序列错误、可用工具未充分利用 | 技能激活提示、技能组合协议、程序支持卡片、小数据 SFT |

### 3.2 分类体系分布

我们初步分析的关键观察：

- 在 PinchBench 上，E 类失败（无效的技能利用）占 23 个任务中的 3-4 个（13-17%）。

- 在 AgentBench-OpenClaw 上，记忆域在所有模型检查点上持续得分 50.0，表明存在结构性 C 类失败，无论训练检查点如何。

- A 类失败出现在 task_15_daily_summary 中，其中 nanobot 初始化创建 AGENTS.md 文件污染工作空间，导致智能体偏离任务目标。

## 4 核心假设体系

我们的研究围绕以下五个核心假设展开，这些假设将我们的失败分类体系从"经验分类"提升为"可验证的机制性结论"。

### H1: 能力-激活鸿沟假说

**Hypothesis H1:** 在 open-world agent benchmark 中，相当比例的失败并非来自 base model 缺乏任务知识，而来自能力到调用之间的激活鸿沟；因此，显式 skill activation / affordance prompting 的收益会显著高于增加模型参数或更晚 checkpoint。

**验证方法：** 同一任务比较三种条件：(1) base，(2) T4b（技能激活提示），(3) 更强checkpoint。若T4b收益接近或超过更强checkpoint，尤其集中在E类，则支撑此假说。进一步拆分E类为E1（available-but-unactivated）和E2（activated-but-miscomposed），预期T4b对E1更强，SFT对E2更强。

### H2: State Abstraction Failure 假说

**Hypothesis H2:** 智能体在 memory tasks 上的主要瓶颈并不是上下文长度或存储容量，而是缺乏对"状态"的显式建模与可操作读取；换言之，失败来自 state abstraction failure，不是单纯的 forgetting。

**验证方法：** 比较三种memory机制：(1) 原始长上下文，(2) 非结构化summary memory，(3) 结构化state tracker（slots: constraints, derived_facts, pending_subgoals, artifact_paths）。若(3)远高于(2)，则说明问题不是容量，而是表示形式。

### H3: 控制闭环断裂假说

**Hypothesis H3:** 很多 agent failures 不是 isolated error types，而是源于同一控制闭环的不同断点：goal grounding → action instantiation → outcome verification。A/B/D 分别对应闭环的三个断裂位置。

**验证方法：** 对每条失败轨迹标记first break point，然后测试planning prompt主要修复前段断裂，preflight/tool grounding修复中段断裂，verifier/reflection修复后段断裂。

### H4: 正交修复机制假说

**Hypothesis H4:** training-free interventions primarily repair control-structure failures, while SFT primarily repairs policy priors for skill selection and composition; therefore their gains are orthogonal rather than monotonic substitutes.

**验证方法：** 构建category-conditioned gain matrix（行：T1/T2/T3/T4/SFT；列：A/B/C/D/E）。预期图像：T2/T3在A/B/D强，T1在C强，SFT/T4a在E强。再测组合增益是否接近additive。

## 5 无训练修复配方

给定分类体系，我们问：*哪些失败类别可以通过推理时配方在无需任何训练的情况下修复？* 我们设计了四种最小化、可解释的配方，每种针对特定的失败类别。

### 5.1 情景记忆 (T1)

**[类别 C] — 验证 H2**

**目标：** 类别 C（记忆/状态管理失败）。

**设计：** 我们为基础智能体配备任务本地情景记忆缓冲区。在每次工具调用后，智能体将所学内容的摘要写入记忆缓冲区。在发出新的工具调用前，智能体读取记忆缓冲区并将相关项目纳入上下文。

### 5.2 单智能体控制配方 (T2)

**[类别 A] [类别 B] [类别 D] — 验证 H3**

**目标：** 类别 A（任务理解/规划漂移）、B（工具/环境落地）和 D（验证缺陷）。

**设计：** 具有四个组件的最小控制配方：(a) 计划优先，(b) 前置检查，(c) 有界重规划，(d) 失败触发反思。

### 5.3 最小协作 (T3)

**[类别 D] [类别 A]**

**目标：** 类别 D（验证）和 A（规划）。

**设计：** 最小双角色结构：执行器调用工具，验证器检查执行器的工具调用及其结果，如果被质疑则要求修改或明确证明。

### 5.4 技能利用 (T4)

**[类别 E] — 验证 H1**

我们为E类设计了两个互补的配方：

#### T4a：程序支持卡片（技能可用性）

**设计：** 对于每个基准测试域，预编译结构化程序支持卡片。在任务开始时，检索最相关的3张卡片并注入系统提示。

#### T4b：技能激活提示（技能利用）

**设计：** 提示策略明确提示智能体 (1) 枚举可用技能，(2) 评估每个技能对当前子任务的相关性，(3) 选择最合适的技能，以及 (4) 验证技能输出。

## 6 小数据 SFT 研究 — 验证 H4

除了无训练配方，我们问：*小数据监督微调可以修复配方无法修复的失败吗？* 我们设计了一个对照研究，在三个数据规模（100、300、1000样本）下使用 LoRA-SFT。

**SFT 设置：** 所有 SFT 变体使用 LoRA rank 16，alpha 32，dropout 0.05，学习率 2×10⁻⁴余弦 schedule，warmup ratio 0.1，batch size 8，最大序列长度 4096 token，2 个 epoch。

**核心研究问题：** (a) SFT 边际价值：与最佳无训练配方相比，SFT 提供了多少额外修复？(b) 数据效率：SFT 修复率 vs. 训练集大小的扩展曲线是什么？(c) 负干扰：SFT 是否会在基础已解决的任务上引入回归？

**[类别 E] task_13_image_gen --- 技能激活失败**

*诊断：* 所有模型在记忆域上持续得分 50.0，无论训练检查点如何。这不是"存不下"的问题，而是"没有把状态变成可操作表示"，支持 H2。

## 8 讨论

### 8.1 核心主张的四个 sharper claims

我们的研究支持以下四条核心主张，这些主张将我们的工作与"工程patchwork"区分开来：

1. **Agent failures are often misdiagnosed as model weakness, while many are actually system-level activation or control failures.** — E类失败不是"模型不会"，而是"有技能但没激活"（H1）

2. **The key distinction is not training-free vs training-based, but structural vs procedural failure.** — 结构性失败（A-D类）用recipe修复，程序性失败（E类）用SFT修复，二者正交（H4）

3. **Memory in agents is not merely storage; it is explicit state representation that must remain decision-relevant across steps.** — 记忆瓶颈不是容量问题，而是state abstraction问题（H2）

4. **Benchmark scores for agents partly conflate reasoning ability with runtime compatibility and interface affordances.** — 部分"A类失败"可能与runtime环境有关，而非纯模型能力问题

### 8.2 假设验证总结

| 假设 | 核心主张 | 支撑证据 |
|------|---------|---------|
| **H1** | E类 = 能力-激活鸿沟 | task_13/20 无论checkpoint都失败；T4b预期有效；E1(E1: available-but-unactivated) vs E2(activated-but-miscomposed)分布待验证 |
| **H2** | C类 = state abstraction failure | 记忆域所有模型卡在50.0，非容量问题 |
| **H3** | A/B/D = 控制闭环断裂 | 断点分布支持（前/中/后段） |
| **H4** | 正交修复机制 | E类需技能激活/A-D需结构干预，SFT补充 |

### 8.2 核心主张的论证

我们的核心主张是 **Open-source agent failures are often structural, not parametric**，具体体现在：

- **E 类：activation gap** — 问题不是"模型不会"，而是"有技能但没激活"（H1）
- **C 类：state abstraction gap** — 问题不是"存不下"，而是"没把状态变成可操作表示"（H2）
- **A/B/D：control-loop breakpoints** — 问题分布在控制闭环的不同断点（H3）
- **方差：runtime compatibility** — benchmark 分数部分来自 runtime 环境因素

### 8.3 对从业者的建议

我们的发现为从业者提供了一个原则性决策树：

1. **先诊断，后修复**：区分"模型不会"还是"系统层失败"
2. **结构性失败用轻量配方**：A/B/D/C 类用 T1-T4（接近零成本）
3. **知识/技能失败用 SFT**：E 类在 SFT 上有最大边际价值
4. **避免过度工程**：不是所有失败都需要重新训练

## 10 结论

我们提出了诊断和修复 OpenClaw 类智能体失败的统一研究。我们的核心主张是 **Open-source agent failures are often structural, not parametric**。我们的贡献跨越三个层次：

1. **评估框架。** 我们构建了可重现的评估协议，固定基准测试版本、任务分割和评估协议，实现跨修复策略的严格配对比较。

2. **失败分类体系。** 我们定义了基于失败案例分析的五类失败分类体系（A-E），识别出跨基准测试一致的结构性失败模式：E 类失败（技能利用缺陷）占 PinchBench 任务的 13-17%；AgentBench-OC 上的记忆域在所有模型检查点上持续得分 50.0。

3. **假设验证框架。** 我们提出并验证了五个核心假设，将论文从"经验报告"提升为"机制性结论"。

**核心发现：** 智能体失败不是均匀随机的，而是聚类到可解释的结构类别中。相当一部分智能体失败并非来自 base model 缺乏任务知识，而来自能力到调用之间的激活鸿沟（E类）、状态表示的抽象失败（C类）、或控制闭环的断裂（A/B/D类）。修复策略必须与失败类型相匹配：training-free 干预修复结构性失败，SFT 修复知识/技能激活失败，二者正交而非替代关系。

## 局限性

1. **初步评估。** 系统修复研究（T1-T5 配方、SFT 条件）的实验评估尚未完成。
2. **单一基础模型。** 所有实验使用 doubao-seed-1.8 模型。
3. **两个基准测试。** 我们的基准测试套件包含两个基准测试。
4. **非正式标注。** 失败分类基于作者案例分析，而非具有标注者间一致性的正式标注。
